# v0.6 Breaking Changes

## Retry errors extend `BaseException`

**What**

Change the control flow errors (`NonRetriableError` and `RetryAfterError`) to extend `BaseException` instead of `Exception`.

**Why**

It’s too easy to accidentally catch these errors. For example, let’s say you want to run some cleanup step when another step errors:

```py
try:
    await ctx.step.run("do-work", do_work)
except Exception:
    await ctx.step.run("rollback", rollblack)
```

If `do-work` raises `NonRetriableError` then we’ll actually _rerun_ it after it errors. This is because `except Exception` catches `NonRetriableError` and swallows it.

If we instead extend `BaseException`, then `except Exception` will not catch it. This is how control flow interruption works in the Python SDK.

## Middleware

Completely rewrite middleware using learnings from TypeScript SDK v4. We'll write details later.

## Logging

### Separate SDK-internal logger from user-facing logger

**Current behavior:** The user-provided `logger` (or the default `logging.getLogger(__name__)`) is used for both app logs (`ctx.logger` in function handlers) and SDK internal logs (connection lifecycle, heartbeats, handshakes, lease extensions, etc.). There are ~60+ internal log calls across `comm_lib/handler.py`, `comm_lib/utils.py`, and the entire `connect/` subsystem — all going through `self._client.logger`, which is the user's logger. Users cannot control SDK log verbosity without affecting their own logs.

**New behavior:** SDK internals use `logging.getLogger("inngest")` with a child logger hierarchy:

- `inngest.connect` — connection lifecycle, heartbeats, handshakes, lease extensions
- `inngest.comm` — HTTP handler, sync requests, signature verification
- `inngest.client` — send retries, mode detection

The user-provided logger is only used for `ctx.logger` in function handlers.

This lets users independently configure SDK log levels:

```python
# Silence noisy Connect logs without losing HTTP handler errors
logging.getLogger("inngest.connect").setLevel(logging.WARNING)

# Or silence all SDK internals
logging.getLogger("inngest").setLevel(logging.WARNING)
```

It also works automatically with structlog, loguru, and any other library that integrates with stdlib logging.

**Default `ctx.logger`:** When the user does not pass a `logger` to the `Inngest` constructor, `ctx.logger` defaults to `logging.getLogger("inngest.functions")` (instead of the current `logging.getLogger(__name__)` which resolves to `inngest._internal.client_lib.client`). This keeps it under the `inngest` namespace but distinct from SDK internals.

### Replace `FilteredLogger` with a `logging.Filter`

**Current behavior:** `FilteredLogger` is a plain class using `__getattr__` to proxy 9 logging methods. It doesn't satisfy `isinstance(x, logging.Logger)` and the `# type: ignore` on `ctx.logger = self._filtered_logger` confirms the type mismatch. The proxy also silently passes through non-logging methods like `setLevel` and `addHandler` without filtering.

**New behavior:** Replace `FilteredLogger` with a stdlib `logging.Filter` that uses the existing `_logging_enabled` ContextVar:

```python
class StepReplayFilter(logging.Filter):
    def filter(self, record: logging.LogRecord) -> bool:
        return _logging_enabled.get()
```

The `LoggerMiddleware.transform_input()` attaches this filter to `ctx.logger` instead of replacing it with a wrapper. This means:

1. `ctx.logger` is never mutated — no `# type: ignore` needed.
2. All log calls are filtered uniformly (not just the 9 hardcoded method names).
3. Works correctly with structlog/loguru that integrate via stdlib handlers.

### Include `exc_info` on SDK-internal error logs

**Current behavior:** SDK error logs discard tracebacks, using patterns like `logger.error("Connection error", extra={"error": str(e)})` throughout the Connect subsystem and HTTP handler.

**New behavior:** Use `exc_info=e` on all SDK-internal error-level logs:

```python
# Before
self._logger.error("Connection error", extra={"error": str(e)})

# After
self._logger.error("Connection error", exc_info=e)
```

This only applies to SDK-internal logs (the `inngest.*` logger hierarchy). User-facing `ctx.logger` calls are entirely under user control — the SDK does not inject `exc_info` into those.

### Remove `set_logger()`

**Current behavior:** `Inngest.set_logger()` allows mutating the logger after construction.

**New behavior:** Remove `set_logger()`. With SDK internals on their own logger hierarchy, `set_logger()` would only affect `ctx.logger` — but `ctx.logger` is set per-request from `self.logger` at context creation time. If users need per-request loggers, middleware's `transform_input` already lets them replace `ctx.logger`. Make `logger` constructor-only.

## Require `datetime.timedelta` for all duration parameters

**Current behavior:** All duration/timeout parameters accept `int | datetime.timedelta`, where `int` means milliseconds. This applies to step methods (`sleep`, `wait_for_event`, `invoke`, `invoke_by_id`), configuration objects (`Batch`, `Cancel`, `Debounce`, `RateLimit`, `Throttle`, `Timeouts`), `Inngest(request_timeout=...)`, and `RetryAfterError(retry_after=...)`.

**Problem:** Python universally uses seconds for numeric time values (`time.sleep(5)`, `asyncio.wait_for(timeout=5)`, `httpx.Timeout(5)`). Inngest's `int = milliseconds` convention causes silent bugs: `step.sleep("id", 5)` sleeps 5ms, not 5s. The test suite already uses `timedelta` almost exclusively, confirming it's the natural choice.

**New behavior:** Drop `int` from all duration types. Only accept `datetime.timedelta`.

```python
# Before
await ctx.step.sleep("delay", 5000)
inngest.RateLimit(limit=5, period=60000)

# After
await ctx.step.sleep("delay", timedelta(seconds=5))
inngest.RateLimit(limit=5, period=timedelta(minutes=1))
```

Users already on `timedelta` need no changes. Users on raw `int` get a type error (not a silent behavior change) and trivially wrap in `timedelta()`.

## Split `Event` into `Event` (send) and `ReceivedEvent` (receive)

**Current behavior:** A single `Event` class is used for both sending and receiving. Optional fields use sentinel defaults (`id: str = ""`, `ts: int = 0`, `data: Mapping[str, JSON] = {}`) that are special-cased during serialization.

**Problems:**

- Sentinel defaults are un-Pythonic and leak into serialization logic (`client.py` strips `id == ""` and replaces `ts == 0`).
- `data` is typed as `Mapping` (read-only) but is actually a `dict` at runtime, causing false type errors on mutation.
- A single class can't express that `id`/`ts` are optional when sending but guaranteed when receiving. Either the send side has confusing defaults, or the receive side forces users to write type guards that never trigger.
- If a user forwards a received event via `client.send(ctx.event)`, the original `id` is preserved and Inngest deduplicates it. This is a silent bug.

**New behavior:** Two independent classes with no inheritance:

```python
class Event(BaseModel):
    """For sending."""
    data: dict[str, Any] = {}
    id: str | None = None
    name: str
    ts: int | None = None

class ReceivedEvent(BaseModel):
    """From the server. All fields guaranteed present."""
    data: dict[str, Any] = {}
    id: str
    name: str
    ts: int

    @pydantic.field_validator("data", mode="before")
    @classmethod
    def _ensure_dict(cls, v: dict[str, Any] | None) -> dict[str, Any]:
        return v or {}
```

Key design decisions:

- **No inheritance.** `send()` accepts `Event` only, not `ReceivedEvent`. This prevents the silent dedupe bug where forwarding `ctx.event` preserves the original `id`. To forward, users must explicitly construct a new `Event`, forcing them to think about which fields to include:
  ```python
  received = ctx.event
  await client.send(Event(name=received.name, data=received.data))
  ```
- **`data` defaults to `{}` on both classes**, not `None`. This preserves the ergonomic `event.data["key"]` access without None guards. The `_ensure_dict` validator lives on `ReceivedEvent` only (the server is the one that may send `null`).
- **`Mapping` becomes `dict`** on both classes. The current `Mapping` type prevents mutation at the type level but `data` is always a `dict` at runtime.

**Public API changes:**

- `ctx.event`, `ctx.events`, and `step.wait_for_event()` return `ReceivedEvent`.
- `client.send()` and `step.send_event()` accept `Event` only.
- `before_send_events` middleware hook takes `list[Event]`.
- Internal: `ServerRequest.event` and `ServerRequest.events` change to `ReceivedEvent`. All `Event.model_validate()` calls on the receive path change to `ReceivedEvent.model_validate()`.

Sending code (`inngest.Event(name=..., data=...)`) requires no changes. Receiving code gains accurate types — `ctx.event.id` is `str`, not `str` that might secretly be `""`.

## Remove the `JSON` type alias from the public API

**Current behavior:** `JSON` is exported as `bool | int | float | str | Mapping[str, object] | Sequence[object] | None` and used as the `Event.data` value type.

**New behavior:** Remove `JSON` from `__all__`. Replace all public uses with `Any`. The `JSON` type was an attempt at a recursive JSON type that couldn't actually be recursive (due to TypedDict compatibility and mypy cycle limitations). `Any` is more honest — untyped data is untyped. Users who want type safety should use `EventType` with a typed `Data` class.

```python
# Before
data: dict[str, JSON]

# After
data: dict[str, Any]
```

## Typed event data via `EventType` and `Context[T]`

**Current behavior:** `ctx.event.data` is `dict[str, Any]`. Users must manually parse or cast event data in every handler. Event names are magic strings with no connection to their data schemas.

**New behavior:** Introduce `EventType` as a schema definition that pairs an event name with a typed data model. `ReceivedEvent` becomes generic over both name and data. `Context` becomes generic over the received event type.

### Core types

```python
NameT = TypeVar("NameT", bound=str)
DataT = TypeVar("DataT", default=dict[str, Any])  # PEP 696 via typing_extensions

class ReceivedEvent(BaseModel, Generic[NameT, DataT]):
    """From the server. All fields guaranteed present."""

    name: NameT
    data: DataT
    id: str
    ts: int
    v: str | None

class EventType:
    """Base class for typed event schemas. Users subclass this."""

    name: ClassVar[str]  # overridden with a Literal type by subclasses

    class Data(BaseModel):
        """Override in subclasses to define the event data shape."""
        ...
```

### Defining event types

Users subclass `EventType` and define a `Received` alias for use in annotations:

```python
class UserCreated(inngest.EventType):
    name: Literal["user.created"] = "user.created"

    class Data(BaseModel):
        email: str
        org_id: str

    Received = ReceivedEvent[Literal["user.created"], Data]


class AdminCreated(inngest.EventType):
    name: Literal["admin.created"] = "admin.created"

    class Data(BaseModel):
        admin_email: str
        org_id: str
        role: str

    Received = ReceivedEvent[Literal["admin.created"], Data]
```

### Single-trigger function

```python
@client.create_function(
    fn_id="process-user",
    trigger=inngest.TriggerEvent(event=UserCreated),
)
async def process_user(ctx: Context[UserCreated.Received]) -> None:
    ctx.event.data.email    # str — fully typed
    ctx.event.name          # Literal["user.created"]
    ctx.event.id            # str
```

### Multi-trigger function with discriminated narrowing

```python
@client.create_function(
    fn_id="onboard",
    trigger=[
        inngest.TriggerEvent(event=UserCreated),
        inngest.TriggerEvent(event=AdminCreated),
    ],
)
async def onboard(
    ctx: Context[UserCreated.Received | AdminCreated.Received],
) -> None:
    # Common fields work without narrowing
    ctx.event.data.org_id   # str — present on both Data types

    # Narrow by event name (standard discriminated union — works in pyright/mypy)
    if ctx.event.name == "user.created":
        ctx.event.data.email        # str — narrowed to UserCreated.Data
    elif ctx.event.name == "admin.created":
        ctx.event.data.admin_email  # str — narrowed to AdminCreated.Data
```

### Sending typed events

`EventType` can provide a helper to construct a send-side `Event`:

```python
await client.send(UserCreated.event(data=UserCreated.Data(email="a@b.com", org_id="org-1")))
# Constructs Event(name="user.created", data={...})
```

Plain `Event(name="foo", data={...})` still works for untyped/dynamic events.

### How it works

- **Registration time:** The SDK extracts the type parameter from the handler's annotation using `typing.get_type_hints()`. For each `EventType` in the trigger list, it builds a name→data type mapping.
- **Runtime:** When the server sends an event, the SDK looks up the event name in the mapping, validates `data` against the corresponding `Data` class using Pydantic, and constructs the parameterized `ReceivedEvent`.
- **`TriggerEvent`** accepts either a string (untyped, backwards compatible) or an `EventType` subclass (typed).

### Unparameterized default

`Context` without a type parameter defaults to `ReceivedEvent[str, dict[str, Any]]` — preserving current behavior. Untyped handlers require no changes.

### `step.wait_for_event`

Accepts a `type[ReceivedEvent]` variant instead of a string, enabling typed returns with no overloads:

```python
ReceivedEventT = TypeVar("ReceivedEventT", bound=ReceivedEvent)

async def wait_for_event(
    self,
    step_id: str,
    *,
    event: type[ReceivedEventT],
    timeout: timedelta,
) -> ReceivedEventT | None: ...
```

Usage:

```python
# Typed — full static safety
event = await ctx.step.wait_for_event("wait", event=UserCreated.Received, timeout=...)
# event: ReceivedEvent[Literal["user.created"], UserCreated.Data] | None

# Untyped — name only, data defaults to dict[str, Any] (via PEP 696 default TypeVar)
event = await ctx.step.wait_for_event("wait", event=ReceivedEvent[Literal["thing.happened"]], timeout=...)
# event: ReceivedEvent[Literal["thing.happened"], dict[str, Any]] | None
```

The SDK extracts the event name from the `ReceivedEvent`'s `NameT` parameter at runtime to subscribe to the correct event.

### Limitations

- Event types (including `Data` classes) must be defined at module scope. Locally-scoped types are strings at runtime (due to PEP 563 / `from __future__ import annotations`) and `typing.get_type_hints()` cannot resolve them.
- Decorated handlers must preserve `__globals__` (e.g. via `@functools.wraps`) for annotation extraction to work. This is standard practice.
- The `Received` alias on each `EventType` subclass is manual boilerplate. It cannot be auto-generated on the base class because `__init_subclass__` assignments are invisible to type checkers.

## Rename `is_production` to `is_dev`

**Current behavior:** `Inngest(is_production=False)` opts into dev server mode. The internal representation is `ServerKind.CLOUD` vs `ServerKind.DEV_SERVER`, but the public API uses a negated boolean.

**Problem:** `is_production=False` is a double negative — users must reason about "not production" to mean "dev server". The parameter name also leaks a deployment concept when the actual semantic is "use the Inngest dev server".

**New behavior:** Replace with `is_dev: bool | None = None`. The default `None` preserves the current auto-detection behavior (env var fallback). Also remove the vestigial `client.is_production` public attribute.

```python
# Before
inngest.Inngest(app_id="my-app", is_production=False)

# After
inngest.Inngest(app_id="my-app", is_dev=True)
```

## Replace `Streaming` enum with a `streaming: bool` kwarg

**Current behavior:** `Streaming` is a public enum (`DISABLE`, `FORCE`) passed to framework `serve()` functions. Only FastAPI supports it; other frameworks silently ignore it.

**New behavior:** Remove the `Streaming` enum from the public API. Replace with `streaming: bool = False` on `fast_api.serve()` only. Other framework `serve()` functions don't get the parameter since they don't support it.

```python
# Before
inngest.fast_api.serve(app, client, functions, streaming=inngest.Streaming.FORCE)

# After
inngest.fast_api.serve(app, client, functions, streaming=True)
```

## Simplify `output_type`

### Problem

Returning non-JSON types (Pydantic models, dataclasses, etc.) requires three pieces of boilerplate:

1. `serializer=inngest.PydanticSerializer()` on the client
2. `output_type=list[User]` on `create_function()` — duplicating the return annotation
3. `output_type=list[User]` on each `step.run()` — duplicating the handler's return annotation

The SDK needs the type to deserialize memoized step output back into the correct Python object. Today that type comes exclusively from `output_type=`. But the user already declares the type in their return annotation — forcing them to write it again is redundant and error-prone.

### Default to `PydanticSerializer`

Make `PydanticSerializer()` the default when no serializer is provided. The `serializer` param stays as an override for custom serialization (ORM models, etc.).

```python
# Before
client = inngest.Inngest(app_id="my-app", serializer=inngest.PydanticSerializer())

# After — just works
client = inngest.Inngest(app_id="my-app")

# Custom serializer still supported
client = inngest.Inngest(app_id="my-app", serializer=MyORMSerializer())
```

### Read output type from return annotation

Instead of requiring a separate `output_type=` parameter, the SDK reads the return type from the handler's annotation via `typing.get_type_hints()`.

**On `create_function`:** At decoration time, extract the return annotation from the handler.

```python
# Before
@client.create_function(fn_id="child", output_type=list[User], trigger=...)
async def fn_child(ctx: inngest.Context) -> list[User]:
    return [User(name="Alice")]

# After — type comes from -> list[User]
@client.create_function(fn_id="child", trigger=...)
async def fn_child(ctx: inngest.Context) -> list[User]:
    return [User(name="Alice")]
```

**On `step.run`:** Extract the return annotation from the step handler.

```python
async def fetch_users() -> list[User]:
    return [User(name="Alice")]

# Before
result = await ctx.step.run("fetch", fetch_users, output_type=list[User])

# After — type comes from -> list[User]
result = await ctx.step.run("fetch", fetch_users)
```

### Type resolution priority

1. **Explicit `output_type=`** — highest priority, always wins
2. **Return type annotation** — read via `typing.get_type_hints(handler).get("return")`
3. **No type known** — serializer is skipped. Output must be plain JSON-serializable. If it's not, error:

```
OutputUnserializableError: Function "my-fn" returned unserializable data.
The output type could not be determined. Either add a return type
annotation to your function, or pass output_type= explicitly.
```

The serializer is only invoked when the output type is known, because deserialization requires the type. Serializing without a known type would produce data that can't be correctly deserialized later.

### Keep `output_type` as explicit override

The parameter stays on both `create_function()` and `step.run()` for cases where annotation reading doesn't work:

- Lambdas without return annotations
- Generic functions (TypeVar return types aren't concrete)
- Decorator chains that drop `__globals__`

### Type extraction utility

```python
def get_output_type(func: Callable[..., Any]) -> object:
    """
    Extract the return type annotation from a callable.
    Returns EmptySentinel if no usable type is found.
    """
    try:
        hints = typing.get_type_hints(func)
    except Exception:
        return EmptySentinel

    ret = hints.get("return")
    if ret is None or ret is type(None):
        return EmptySentinel
    if isinstance(ret, TypeVar):
        return EmptySentinel
    return ret
```

### Edge cases

| Case                                 | Behavior                                                                          |
| ------------------------------------ | --------------------------------------------------------------------------------- |
| No return annotation                 | EmptySentinel → serializer skipped → must be plain JSON-serializable              |
| Returns `None`                       | EmptySentinel → no serialization needed                                           |
| Returns `str` / `int` / `bool`       | Type extracted → TypeAdapter passes through as-is                                 |
| Returns `dict[str, User]`            | Type extracted → serializer handles nested Pydantic models                        |
| Returns `Optional[User]`             | Type extracted → serializer round-trips correctly                                 |
| Returns `list[User]`                 | Type extracted → serializer round-trips correctly                                 |
| TypeVar return (`T`)                 | EmptySentinel → must use explicit `output_type=`                                  |
| Lambda (no annotation)               | EmptySentinel → must use explicit `output_type=`                                  |
| `from __future__ import annotations` | `get_type_hints()` resolves via `func.__globals__` — works for module-scope types |
| Explicit `output_type=` set          | Overrides annotation                                                              |
| Custom `serializer` on client        | Used instead of PydanticSerializer                                                |

### Limitations

- Types must be defined at module scope when using `from __future__ import annotations` (same constraint as the `EventType` proposal). `typing.get_type_hints()` resolves stringified annotations via `func.__globals__`.
- Decorated handlers must preserve `__globals__` (e.g. via `@functools.wraps`).
- Generic functions with TypeVar returns can't be resolved — must use explicit `output_type=`.

### Full example

```python
from pydantic import BaseModel
import inngest

class User(BaseModel):
    name: str

client = inngest.Inngest(app_id="my-app")

@client.create_function(
    fn_id="child",
    trigger=inngest.TriggerEvent(event="never"),
)
async def fn_child(ctx: inngest.Context) -> list[User]:
    return [User(name="Alice")]

@client.create_function(
    fn_id="parent",
    trigger=inngest.TriggerEvent(event="test"),
)
async def fn_parent(ctx: inngest.Context) -> list[User]:
    # step.invoke — type from Function[list[User]] (unchanged)
    output = await ctx.step.invoke("invoke", function=fn_child)

    # step.run — type from handler's -> list[User] annotation
    async def fetch() -> list[User]:
        return [User(name="Bob")]
    result = await ctx.step.run("fetch", fetch)

    # step.run with lambda — explicit output_type required
    result2 = await ctx.step.run(
        "fetch2",
        lambda: [User(name="Charlie")],
        output_type=list[User],
    )

    return output + result + result2
```

## Things to ignore

I will not stop using `MaybeError`. Don't suggest it.

`step.run` accepting variadic `*handler_args`.

Merging `ctx.group` into `ctx.step` or changing the `parallel()` API.

Unifying `send`/`send_sync` into a single method.

Renaming `fn_id` to `id` in `create_function()`.

Ignore the `to_duration_str` bug.

Make `step_id` keyword-only with smart defaults on all step methods except `step.run`.
